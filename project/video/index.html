The problem

Transcription is easy to start and hard to make dependable.

The product was designed around real failure modes: missing captions, platform restrictions, long-running local work, limited memory, and results that lose their source context.

01 / Availability

Not every video exposes text

The tool inspects available sources before promising an outcome, then offers a local fallback when downloadable audio is available.

02 / Trust

Long tasks can look broken

Visible stages, elapsed time, progress percentage, cancellation, and actionable errors keep the user informed and in control.

03 / Quality

Raw captions are difficult to read

Rolling duplicates are cleaned into readable paragraphs while timestamped and raw diagnostic views remain available.

A deliberate workflow

Use the lightest reliable path.

01

Inspect

Validate the public URL, identify the platform, read safe metadata, check duration, and discover available transcript sources.

02

Select

Prefer creator or automatic captions. Use public post text when it is the useful source. Download audio only for local transcription.

03

Save

Create a readable transcript, retain source information, provide timestamped exports, and remove temporary media.

Product decisions

Constraints became part of the design.

This is a focused personal tool, not a generic cloud transcription service. That boundary shaped its technical choices and interface.

Hardware aware

Safe models for an 8 GB Apple Silicon Mac

The interface prevents unsupported model choices and shows first-use model size, disk readiness, and download progress.

Failure aware

Recovery paths instead of raw errors

Known platform failures lead to specific next actions, while diagnostics omit cookies, signed media URLs, and private data.

Source aware

Different text sources stay different

Captions, public post text, and Whisper transcripts are explicitly labelled rather than presented as interchangeable results.

Lifecycle aware

Start, stop, cancel, and clean up safely

Managed processes prevent duplicate launches, cancellation stops real work, and partial media is deleted after failure or completion.

System architecture

A public case study. A private working system.

The showcase is static and always online. The operational workspace is protected separately and keeps processing, transcripts, and temporary media on the owner's Mac.

Scope and honesty

Clear about what it can—and cannot—do.

Platform extraction changes over time. The product communicates availability before processing and avoids promising universal access.

Designed to support

  • Public YouTube videos and available caption tracks
  • Public Facebook videos, Reels, and substantial post text
  • Public Instagram Reels with downloadable media
  • Local audio or video file transcription
  • English, Mandarin, and Cantonese workflows

Deliberate constraints

  • No private-content or access-control bypass
  • No permanent downloaded-media library
  • No public transcript library or public upload endpoint
  • No guarantee that every platform URL remains extractable
  • Live owner workspace is unavailable while the Mac is offline
